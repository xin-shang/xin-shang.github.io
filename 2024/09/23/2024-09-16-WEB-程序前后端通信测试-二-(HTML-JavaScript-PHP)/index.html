流程

PHP 文件架构

首先对后端架构进行整体设计，以下是项目文件结构：

project/
├── public/                    // 公共访问目录，入口文件
│   └── index.php
├── app/                       // 应用核心代码
│   ├── controllers/           // 控制器
│   │   └── UserController.php
│   ├── models/                // 数据模型
│   │   └── User.php
│   ├── core/                  // 核心类
│   │   ├── Router.php         // 路由类
│   │   ├── Request.php        // 请求类
│   │   ├── Response.php       // 响应类
│   │   ├── Controller.php     // 基础控制器
│   │   └── Model.php          // 基础模型
│   └── routes.php             // 路由定义文件
├── config/                    // 配置文件
│   └── config.php
├── vendor/                    // 第三方库（可选，用于自动加载）
│   └── autoload.php
├── logs/                      // 日志文件
├── tests/                     // 测试代码
└── composer.json              // Composer 配置文件（可选）

说明：

  • public/：存放入口文件 index.php，所有请求都将通过此文件。
  • app/：核心应用程序代码。
  • controllers/：控制器，处理请求并返回响应。
  • models/：数据模型，处理数据库交互和业务逻辑。
  • core/：核心类，包括路由、请求、响应、基础控制器和模型等。
  • routes.php：定义应用的路由规则。
  • config/：配置文件，如数据库连接信息等。
  • vendor/：使用 Composer 自动加载类（即使不使用第三方库，也建议使用 Composer 来管理自动加载）。
  • logs/：存放日志文件。
  • tests/：测试代码，确保代码质量。
  • composer.json：Composer 配置文件，用于管理依赖和自动加载。

程序架构

采用简化的 MVC（模型-视图-控制器）架构，实现代码的模块化和可维护性。

  • 路由器（Router）：解析请求的 URL 和方法，分发到相应的控制器和方法。

  • 控制器（Controller）：处理请求，调用模型获取数据，返回响应。

  • 模型（Model）：与数据库交互，处理业务逻辑。

  • 请求和响应（Request & Response）：封装 HTTP 请求和响应，方便数据的获取和返回。

运行逻辑和整体流程

请求流程

  1. 客户端请求：前端（Web、移动端等）向后端 API 发起 HTTP 请求，例如 GET /api/users/1。
  2. 入口文件：请求到达 public/index.php，作为应用的唯一入口。
  3. 自动加载：使用 Composer 的自动加载功能，引入所需的类。
  4. 初始化路由器：Router 类解析请求的路径和方法。
  5. 路由分发：根据路由规则，路由器调用对应的控制器方法。
  6. 控制器处理：控制器接收请求，调用模型获取或操作数据。
  7. 模型交互：模型与数据库交互，执行相应的查询或操作。
  8. 响应返回：控制器将数据封装为 JSON 格式，使用 Response 类返回给客户端。

请求类型示例

  • 浏览数据（GET）：GET /api/users，获取用户列表。
  • 获取单个数据（GET）：GET /api/users/{id}，获取指定用户信息。
  • 添加数据（POST）：POST /api/users，创建新用户。
  • 更新数据（PUT）：PUT /api/users/{id}，更新指定用户信息。
  • 删除数据（DELETE）：DELETE /api/users/{id}，删除指定用户。

代码规范和最佳实践

  • 遵循 PSR 标准：遵循 PHP-FIG 的 PSR-1 和 PSR-12 编码标准，保持代码风格一致。
  • 自动加载：使用 Composer 的 PSR-4 自动加载规范，方便类的加载和管理。
  • 命名空间：使用命名空间组织代码，与文件结构保持一致。
  • 错误处理：统一的错误和异常处理，返回有意义的 HTTP 状态码和错误信息。
  • 安全性：防范 SQL 注入、XSS 等常见安全问题，使用预处理语句和适当的输入验证。
  • 配置管理：将可变的配置项（如数据库连接信息）放在配置文件中，方便管理和维护。

留出可扩展空间

  • 模块化设计：代码按照功能模块组织，方便未来添加新功能或模块。
  • 中间件支持：设计路由器时考虑中间件机制，方便添加认证、日志等功能。
  • 环境配置：支持多环境配置（开发、测试、生产），方便应用部署和维护。
  • 日志记录：完善的日志机制，方便调试和错误追踪。
  • 单元测试：编写测试代码，确保代码质量和稳定性。

app/core/Router.php

<?php
        // 路由类
        namespace App\core;

        class Router {
                protected $routes = [];

                /**
                 * 将请求方法、URI 和对应的处理程序添加到路由表中。
                 * compact() 是 PHP 的内置函数，用于创建一个包含变量名和变量值的数组。
                 */
                public function add($method, $uri, $action) {
                        $this -> routes[] = compact('method', 'uri', 'action');
                }

                /**
                 * 接收请求对象，根据请求的方法和路径，寻找匹配的路由规则。
                 * 如果找到匹配的路由，调用对应的处理程序；否则，返回默认响应。
                 */
                public function dispatch (Request $request) {
                        $requestMethod = $request -> getMethod();
                        $requestPath = $request -> getPath();

                        // 遍历路由规则
                        foreach($this -> routes as $route) {
                                if($this -> match($route, $requestMethod, $requestPath)) {
                                        return $this -> callAction($route['action'], $request);
                                }
                        }

                        // 未匹配到路由，返回默认响应
                        header('Content-Type: application/json');
                        
                        $response = [
                                'message' => 'Request received successfully!',
                                'method' => $requestMethod,
                                'data' => $request->getJson(),
                        ];

                        echo json_encode($response);
                }

                /**
                 * 判断给定的路由规则是否与当前请求的方法和路径匹配。
                 *
                 * @param [type] $routes
                 * @param [type] $requestMethod
                 * @param [type] $requestPath
                 * @return void
                 */
                protected function match ($route, $requestMethod, $requestPath) {
                        // 如果路由规则中的方法与请求的方法不一致，返回 false，表示不匹配。
                        if($route['method'] !== $requestMethod) {
                                return false;
                        }
                        // 如果方法匹配，比较路由规则中的 URI 与请求的路径。
                        return $route['uri'] === $requestPath;
                }

                /**
                 * 根据路由规则中指定的处理程序，执行相应的操作。
                 * 处理程序可以是回调函数，也可以是一个 PHP 文件。
                 *
                 * @param [type] $action
                 * @param [type] $request
                 * @return void
                 */
                protected function callAction($action, $request) {
                        // 检查 $action 是否是可调用。正常网址访问？
                        if(is_callable($action)) {
                                return call_user_func($action, $request);
                        } 
                        // 如果 $action 不是可调用的，则认为它可能是一个文件名。
                        elseif (file_exists(__DIR__ . '/../controllers/' . $action)) {
                                // 向上两级，到 doc 目录下的指定文件。
                                require __DIR__ . '/../controllers/' . $action;
                                return;
                        }
                        // 处理程序不存在。
                        else{
                                // 找不到处理程序，返回 404
                                header('Content-Type: application/json');
                                http_response_code(404);
                                echo json_encode(['error' => 'Not Found']);
                        }
                }
        }
?>

app/coreRequest.php

<?php
        /**
         * 构造类 - 请求
         */              
        namespace App\core;

        class Request {

                protected $method;
                protected $path;
                protected $body;
                protected $queryParams;

                /**
                 * 这是一个静态工厂方法，用于从全局变量创建 Request 实例。
                 * @return Request
                 */
                public static function createFromGlobals() {
                        $method = $_SERVER['REQUEST_METHOD'];
                        $path = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
                        $body = file_get_contents('php://input');
                        $queryParams = $_GET;

                        return new self($method, $path, $body, $queryParams);
                }

                /**
                 * 构造函数，初始化请求对象的属性。
                 * @param [type] $method        请求方法
                 * @param [type] $path          请求路径
                 * @param [type] $body          请求体
                 * @param array $queryParams    参数信息
                 */
                public function __construct($method, $path, $body, $queryParams = []) {
                        $this->method = $method;
                        $this->path = $path;
                        $this->body = $body;
                        $this->queryParams = $queryParams;
                }

                      // 
                public function getMethod() {
                        return $this->method;
                }

                      //
                public function getPath() {
                        return $this->path;
                }

                      // 
                public function getJson() {
                        return json_decode($this->body, true);
                }

                      // 
                public function getQueryParams() {
                        return $this->queryParams;
                }
        }
?>

Compose

Composer 是一个 PHP 的依赖管理工具，用于管理和安装项目中的库和依赖项。它可以让开发者定义项目中所需的外部库（dependencies），并自动下载、安装这些库以及它们的依赖项。Composer 通过一个名为 composer.json 的配置文件来管理依赖项的信息，并根据该文件自动加载项目中需要的类。

主要功能

  • 依赖管理：可以为项目添加第三方库，并自动解决这些库之间的依赖关系。**
  • 自动加载：Composer 可以根据配置文件中的设置，自动加载项目中的类，避免手动引入文件。**
  • 版本管理：Composer 可以根据库的版本号进行版本控制，确保安装合适的版本。

自动加载原理

Composer 通过在项目根目录下生成一个 vendor/autoload.php 文件，来实现自动加载。当需要加载某个类时，Composer 会根据类的命名空间自动找到相应的文件并引入，从而不需要手动 require 每个文件。

使用步骤

下载 Composer 安装程序

在终端中运行以下命令来下载 Composer 安装程序：

php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"

注：在任意路径下运行下载 Composer 安装程序的命令是可以的，因为下载的文件（composer-setup.php）只是临时的安装脚本，后续你会将 Composer 安装到系统的全局路径中。完成后，你可以删除下载的 composer-setup.php 文件。

验证安装程序（可选）

为了确保下载的安装程序没有损坏或被篡改，可以运行以下命令来验证安装程序的 SHA-384 散列值：

php -r "if (hash_file('sha384', 'composer-setup.php') === 'a5c698ffe4b8e2b0d708d23c5a6eea3e15e7e037baf01d7a4d455eef2f2e80cbe7a89b56bc94be8b74323892d26d5e1d') { echo 'Installer verified'; } else { echo 'Installer corrupt'; unlink('composer-setup.php'); } echo PHP_EOL;"

安装 Composer

验证完成后，运行以下命令来安装 Composer：

php composer-setup.php

注(两步后)：安装完成后，你需要将 composer.phar 文件移动到全局路径中，这样你可以在系统的任意目录中使用 composer 命令。

删除安装脚本

安装完成后，删除安装脚本：

php -r "unlink('composer-setup.php');"

将 Composer 移动到全局可执行位置

将 Composer 移动到系统的全局可执行目录，使其在任何地方都可以使用：

sudo mv composer.phar /usr/local/bin/composer

不需要将 composer.phar 移动到 PHP 的安装路径中，将它放在 /usr/local/bin/ 目录下即可实现全局可执行。

验证 Composer 安装

你可以通过以下命令来验证 Composer 是否已成功安装：

composer --version

如果一切正常，你应该会看到 Composer 的版本信息。此后，你可以在项目中使用 composer init 来初始化项目。

2. 使用 php 自带的 stream_copy_to_stream

如果仍然遇到连接问题，可以使用 PHP 自带的文件下载方法：

3. 检查网络连接

由于下载失败可能与网络问题有关，你可以尝试以下操作：

  • 确认网络连接是否稳定。
  • 尝试关闭 VPN 或代理服务器，如果你在使用这些工具。

如果问题仍然没有解决，可能需要更新你的 curl 或者系统的 SSL 库（如 LibreSSL）。你可以通过 Homebrew 安装和更新 curl：

brew install curl
brew upgrade curl

更新完成后，重新尝试下载 Composer。

应用方法

Composer 已成功安装并可以全局使用！你现在可以在任何目录下运行 composer 命令来管理 PHP 项目的依赖了。

接下来，如果你需要初始化 Composer 或管理项目中的依赖，可以使用以下命令：

初始化项目

初始化一个项目的 composer.json

composer init
安装依赖项

根据 composer.json 安装依赖项

composer install
更新依赖项
composer update

这是一些简单的命令，以后会专门写有关 Compose 的使用方法的文章。
